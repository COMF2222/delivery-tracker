# Delivery Tracker

Backend-сервис для отслеживания посылок, написанный на Go.

## Возможности

### Посылки

* Создание посылки
* Генерация уникального трек-номера
* Получение информации о посылке по трек-номеру
* Получение списка посылок
* Пагинация списка посылок
* Фильтрация списка по статусу
* Архивация доставленных посылок

### Статусы посылок

Поддерживаются следующие статусы:

* CREATED
* PURCHASED
* WAREHOUSE
* IN_TRANSIT
* CUSTOMS
* ARRIVED
* DELIVERED

Реализована проверка переходов между статусами.

Поддерживается только последовательный переход:

```text
CREATED
    ↓
PURCHASED
    ↓
WAREHOUSE
    ↓
IN_TRANSIT
    ↓
CUSTOMS
    ↓
ARRIVED
    ↓
DELIVERED
```

Пропуск статусов и откат назад запрещены.

### История изменений

При каждом изменении статуса автоматически создаётся запись в истории.

Для каждой записи сохраняются:

* предыдущий статус
* новый статус
* местоположение
* пользователь, выполнивший изменение
* дата изменения

### Фотографии посылок

Реализована загрузка фотографий через `multipart/form-data`.

Поддерживается:

* загрузка файлов
* сохранение файлов на диск
* сохранение пути к файлу в PostgreSQL
* ограничение размера файла
* проверка расширения файла (`jpg`, `jpeg`, `png`)

### Пользователи

Реализована работа с пользователями:

* создание пользователя
* получение информации о текущем пользователе
* деактивация пользователя

Поддерживаются роли:

* admin
* manager

### Аутентификация и авторизация

Реализовано:

* хеширование паролей (`bcrypt`)
* JWT-аутентификация
* middleware для проверки JWT
* ролевая модель доступа (RBAC)
* передача информации о пользователе через `context.Context`

### Аудит действий

Фиксируются следующие действия пользователей:

* изменение статуса посылки
* архивация посылки
* деактивация пользователя

Для каждого действия сохраняются:

* пользователь
* тип действия
* старое значение
* новое значение
* сущность, над которой выполнено действие
* время выполнения

### Транзакции

Следующие операции выполняются в рамках одной транзакции:

* изменение статуса + история + аудит
* архивация посылки + аудит
* деактивация пользователя + аудит

### Кэширование

Для ускорения получения информации о посылке используется Redis.

Используется стратегия **Cache Aside**.

Алгоритм работы:

1. Проверка данных в Redis.
2. Если данные найдены — возвращаются из кэша.
3. Если данных нет — выполняется запрос в PostgreSQL.
4. Полученный результат сохраняется в Redis.
5. При изменении статуса, архивировании или добавлении фотографии соответствующий кэш автоматически инвалидируется.

### Тестирование

Реализованы unit-тесты бизнес-логики сервисного слоя.

Покрыты сценарии:

* cache hit
* cache miss
* ошибка репозитория
* ошибка получения фотографий
* ошибка получения истории
* ошибка записи в Redis

### Swagger

Документация API генерируется автоматически с помощью Swagger.

После запуска приложения интерфейс доступен по адресу:

```text
http://localhost:8081/swagger/index.html
```

# Запуск проекта

## 1. Клонировать репозиторий

```bash
git clone <repository_url>
cd delivery-tracker
```

## 2. Создать `.env`

Создать файл `.env` на основе `.env.example`.

## 3. Запустить инфраструктуру

```bash
docker compose up -d
```

Будут запущены:

* PostgreSQL
* Redis

## 4. Запустить приложение

```bash
go run ./cmd/api
```

# API

## Авторизация

```http
POST /api/v1/auth/login
```

## Пользователи

```http
POST /api/v1/users

GET /api/v1/users/me

PATCH /api/v1/users/deactivate?id={id}
```

## Посылки

```http
GET /api/v1/parcels?page={page}&limit={limit}&status={status}

POST /api/v1/parcels

GET /api/v1/parcels/track?track_number={track_number}

PATCH /api/v1/parcels/status?id={id}

POST /api/v1/parcels/photos?id={id}

PATCH /api/v1/parcels/archive?id={id}
```
